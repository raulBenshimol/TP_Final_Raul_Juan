📚 Capítulo 1: Conceptos Básicos (Desde Cero)

🧠 ¿Qué es un Backend?

Imagina un restaurante:

  • El mesero = El Frontend (Angular). Toma tu pedido y te trae el plato.
  • El chef = El Backend (NestJS). Cocina los ingredientes y prepara el plato.
  • La despensa = La Base de Datos / API externa. Donde están los ingredientes crudos.

El backend nunca habla directamente con el cliente (usuario). Solo habla con el mesero (frontend).

🧩 ¿Qué es NestJS?

NestJS es un framework (conjunto de herramientas) para construir backends en TypeScript.

Organiza el código en tres piezas fundamentales:

Pieza Analogía Responsabilidad
Controller El capitán del equipo Recibe las peticiones HTTP y las delega
Service El chef que cocina Contiene toda la lógica de negocio
Module La cocina (área de trabajo) Agrupa controllers y services relacionados

📦 ¿Qué es una Interface?

Una interface es como un molde de galletas:

  • Define la forma que deben tener los datos
  • No contiene lógica, solo describe estructura
  • Ejemplo: "Un partido DEBE tener: stage, matchCode, goalsFor..."

🏗️ ¿Qué es una Clase?

Una clase es como una fábrica de objetos:

  • Puede crear objetos con new
  • Puede contener lógica en su constructor
  • Ejemplo: new JourneyScreenModel({...}) crea un objeto procesado

🌐 ¿Qué es Swagger?

Swagger es un menú interactivo de tu restaurante:

  • Muestra todos los "platos" (endpoints) disponibles
  • Permite probar cada endpoint sin necesidad de frontend
  • Muestra qué parámetros necesita cada endpoint
  • Muestra la respuesta esperada

📄 Archivo 1: journey-api.interface.ts (CREADO)

📍 Ubicación: src/world-cup/services/model/journey-api.interface.ts

🎯 Propósito: Describir la forma exacta de los datos que nos manda la WorldCupAPI

🧠 Analogía: El Formulario del Proveedor

Cuando pides ingredientes a la despensa central (WorldCupAPI), necesitas saber exactamente cómo te los van a entregar. Esta interfaz es ese "formulario" que describe el paquete que recibirás.

// Esto describe UN partido en el camino del equipo
export interface JourneyMatchApiItem {
  stage: string;           // Ej: "GROUP_STAGE", "QUARTER_FINALS", "FINAL"
  matchCode: string;       // Ej: "A-M1", "QF-1"
  opponentTeamId: string;  // Ej: "jpn", "bra"
  opponentTeamName: string;// Ej: "Japan", "Brazil"
  goalsFor: number;        // Goles a favor (ej: 2)
  goalsAgainst: number;    // Goles en contra (ej: 1)
  result: 'WIN' | 'LOSS' | 'DRAW';  // Solo puede ser uno de estos 3
  resolution: string;      // Ej: "REGULAR_TIME", "PENALTIES"
  isPending: boolean;      // true = partido no jugado aún
}

// Esto describe la respuesta COMPLETA del endpoint
export interface JourneyApiResponse {
  worldCupId: string;           // ID único del mundial
  teamId: string;               // ID del equipo (arg, aus, bra...)
  teamName: string;             // Nombre del equipo
  worldCupStatus: string;       // Estado del mundial
  stageReached: string;         // Hasta dónde llegó el equipo
  isChampion: boolean;          // ¿Fue campeón?
  isFinalPending: boolean;      // ¿Falta jugar la final?
  eliminatedByTeamId: string | null;    // Quién lo eliminó (o null)
  eliminatedByTeamName: string | null;  // Nombre del eliminador
  summary: string;              // Texto narrativo en español
  matches: JourneyMatchApiItem[]; // Lista de partidos
}

🔍 Explicación línea por línea

  • export interface → Hace que esta interfaz pueda usarse desde otros archivos
  • stage: string → El campo "stage" debe ser texto (ej: "GROUP_STAGE")
  • goalsFor: number → El campo "goalsFor" debe ser número (ej: 2)
  • isPending: boolean → Solo puede ser true o false
  • result: 'WIN' | 'LOSS' | 'DRAW' → Solo acepta estos 3 valores exactos
  • eliminatedByTeamId: string | null → Puede ser texto O nulo
  • matches: JourneyMatchApiItem[] → Una lista/array de partidos

⚠️ ¿Por qué NO usamos `?` en los campos?

En otras interfaces del proyecto usan campo?: string (opcional). Nosotros NO usamos ? porque los datos del endpoint de journey siempre vienen completos. Esto nos da más seguridad.

📄 Archivo 2: journey-view-model.ts (CREADO)

📍 Ubicación: src/world-cup/services/model/journey-view-model.ts

🎯 Propósito: Definir cómo entregamos los datos al frontend, ya procesados y en español

🧠 Analogía: El Plato Emplatado

Los ingredientes crudos (API) no se sirven directamente al cliente. El chef los cocina, los decora y los emplata. Este archivo define el "plato terminado" que el mesero (frontend) llevará a la mesa.

// Traducción de resultados a español
const RESULT_LABELS: Record<string, string> = {
  WIN: 'VICTORIA',
  LOSS: 'DERROTA',
  DRAW: 'EMPATE',
};

// Un partido procesado, listo para mostrar
export class JourneyMatchViewModel {
  public stage: string;
  public matchCode: string;
  public opponentTeamId: string;
  public opponentTeamName: string;
  public goalsFor: number;
  public goalsAgainst: number;
  public score: string;        // ← NUEVO: "2 - 1" o "Pendiente"
  public result: string;       // ← TRADUCIDO: "VICTORIA" en vez de "WIN"
  public resultClass: string;  // ← NUEVO: "win", "loss", "draw" (para CSS)
  public resolution: string;
  public isPending: boolean;

  constructor(match: { ... }) {
    // Asigna cada campo
    this.stage = match.stage;
    // ... más asignaciones ...
    
    // LÓGICA: Si está pendiente, muestra "Pendiente"
    this.score = match.isPending
      ? 'Pendiente'
      : `${match.goalsFor} - ${match.goalsAgainst}`;
    
    // LÓGICA: Traduce WIN → VICTORIA
    this.result = RESULT_LABELS[match.result] || match.result;
    
    // LÓGICA: Clase CSS para colores
    this.resultClass = match.result.toLowerCase();
  }
}

// La pantalla completa
export class JourneyScreenModel {
  public teamId: string;
  public teamName: string;
  public stageReached: string;
  public summary: string;
  public totalMatches: number;     // ← NUEVO: Calculado
  public eliminatedBy: string | null; // ← NUEVO: Texto formateado
  public matches: JourneyMatchViewModel[];
  
  constructor(data: { ... }) {
    // Asigna cada campo
    this.teamId = data.teamId;
    // ... más asignaciones ...
    
    // LÓGICA: Cuenta los partidos
    this.totalMatches = data.matches.length;
    
    // LÓGICA: Formatea el mensaje de eliminación
    this.eliminatedBy = data.eliminatedByTeamName
      ? `Eliminado por: ${data.eliminatedByTeamName}`
      : null;
  }
}

🔍 Transformaciones que hace este archivo

Dato de API (crudo) Dato procesado ¿Dónde se hace?
result: "WIN" result: "VICTORIA" RESULT_LABELS[match.result]
goalsFor: 2, goalsAgainst: 1 score: "2 - 1" ${goalsFor} - ${goalsAgainst}
isPending: true score: "Pendiente" Operador ternario ?:
result: "WIN" resultClass: "win" .toLowerCase()
eliminatedByTeamName: "Uruguay" eliminatedBy: "Eliminado por: Uruguay" Template string
matches: [3 partidos] totalMatches: 3 matches.length

⚠️ ¿Por qué usamos CLASES y no solo interfaces?

Las interfaces solo describen estructura. Las clases pueden tener un constructor que ejecuta lógica al crear el objeto. Así transformamos datos automáticamente al hacer new JourneyMatchViewModel(match).

Además, seguimos el mismo patrón que TeamHistoryScreenModel (consistencia con el proyecto).

📍 Ubicación: src/world-cup/services/journey.service.ts

🎯 Propósito: Contiene TODA la lógica de negocio de "Camino del Equipo"

🧠 Analogía: El Chef Principal

Este es el chef que recibe la orden, va a la despensa, cocina los ingredientes, emplata el plato y se lo entrega al capitán (Controller).

// ==========================================
// MAPAS DE ERROR: Qué hacer cuando la API falla
// ==========================================

const JOURNEY_API_ERROR_STATUS_MAP: ApiErrorStatusMap = {
  // Si la WorldCupAPI responde 404 (no encontrado)
  [HttpStatus.NOT_FOUND]: {
    messageCode: WorldCupCoreErrorCode.WC_JOURNEY_UNAVAILABLE,
    message: 'Journey data for the selected team was not found.',
  },
  // Si la WorldCupAPI responde 409 (conflicto - no hay simulación)
  [HttpStatus.CONFLICT]: {
    messageCode: WorldCupCoreErrorCode.WC_JOURNEY_UNAVAILABLE,
    message: 'Journey is not available right now.',
  },
};

// Si falla por cualquier otra razón
const JOURNEY_API_ERROR_FALLBACK: ApiErrorMappingRule = {
  messageCode: WorldCupCoreErrorCode.WC_JOURNEY_UNAVAILABLE,
  message: 'Unable to load journey from World Cup API.',
  statusCode: HttpStatus.BAD_GATEWAY,  // 502
};

// ==========================================
// EL SERVICIO PRINCIPAL
// ==========================================

@Injectable()  // ← Dice "soy un servicio que otros pueden usar"
export class JourneyService extends AbstractBaseService {
  
  constructor(
    private readonly worldCupFeatureApiService: WorldCupFeatureApiService,
    adminService: AdminService,
  ) {
    super(adminService);  // Llama al constructor del padre
  }

  public async getJourney(lang?: string): Promise<JourneyScreenModel> {
    try {
      // PASO 1: Leer el TEAM_ID del .env (sin hardcodear)
      const teamId = this.getCurrentTeamId();
      
      // PASO 2: Normalizar el idioma (si no viene, usa APP_LANG del .env)
      const resolvedLang = this.resolveLang(lang);
      
      // PASO 3: Llamar a la WorldCupAPI para obtener datos crudos
      const apiResponse: JourneyApiResponse =
        await this.worldCupFeatureApiService.getTeamJourney(teamId, resolvedLang);
      
      // PASO 4: Transformar cada partido al formato de vista
      const matches: JourneyMatchViewModel[] = apiResponse.matches.map(
        (match) => new JourneyMatchViewModel(match),
      );
      
      // PASO 5: Construir el modelo de pantalla completo
      return new JourneyScreenModel({
        teamId: apiResponse.teamId,
        teamName: apiResponse.teamName,
        worldCupId: apiResponse.worldCupId,
        worldCupStatus: apiResponse.worldCupStatus,
        stageReached: apiResponse.stageReached,
        isChampion: apiResponse.isChampion,
        isFinalPending: apiResponse.isFinalPending,
        summary: apiResponse.summary,
        eliminatedByTeamName: apiResponse.eliminatedByTeamName,
        matches,
      });
      
    } catch (error) {
      // PASO 6: Si algo falla, usar los mapas de error definidos arriba
      ErrorUtils.mapWorldCupApiError(
        error,
        JOURNEY_API_ERROR_STATUS_MAP,
        JOURNEY_API_ERROR_FALLBACK,
      );
    }
  }
}

🔍 Explicación del flujo interno

  1. getCurrentTeamId() → Lee TEAM_ID=aus del .env. NUNCA está hardcodeado.
  2. resolveLang('es') → Si el frontend manda ?lang=es, lo usa. Si no, usa APP_LANG=es del .env.
  3. getTeamJourney(teamId, resolvedLang) → Va a la despensa (WorldCupAPI) a buscar los ingredientes.
  4. .map((match) => new JourneyMatchViewModel(match)) → Transforma CADA partido. .map() es como decir "para cada elemento, aplica esta transformación".
  5. new JourneyScreenModel({...}) → Crea el plato emplatado con todos los datos procesados.
  6. catch (error) → Si algo falla en cualquier paso, ErrorUtils convierte el error en una respuesta estructurada que el frontend entiende.

🔗 ¿Cómo se acopla con lo existente?

  • Extiende AbstractBaseService → Hereda getCurrentTeamId() y resolveLang()
  • Usa WorldCupFeatureApiService → Para llamar a la API
  • Usa WorldCupCoreErrorCode → Para los códigos de error
  • Usa ErrorUtils → Para manejar errores (ya existía)
  • Sigue el patrón de TeamHistoryService

📄 Archivo 6: journey.controller.ts (MODIFICADO)

📍 Ubicación: src/world-cup/controllers/journey.controller.ts

🧠 Analogía: El Capitán del Equipo

El Controller es el que recibe la orden del mesero (frontend) y se la pasa al chef (Service). No cocina, solo coordina.

import { Controller, Get, Query } from '@nestjs/common';
import { ApiOperation, ApiQuery, ApiResponse, ApiTags } from '@nestjs/swagger';
import { JourneyService } from '../services/journey.service';
import { JourneyScreenModel } from '../services/model/journey-view-model';

@ApiTags('world-cup')           // Agrupa en Swagger bajo "world-cup"
@Controller('world-cup')        // URL base: /worldCupCore/world-cup
export class JourneyController {
  
  // Inyecta el JourneyService (el chef)
  constructor(private readonly journeyService: JourneyService) {}

  @Get('team-journey')          // URL completa: GET /worldCupCore/world-cup/team-journey
  @ApiOperation({ summary: 'Get the full world cup journey for the configured team' })
  @ApiQuery({ name: 'lang', required: false, description: 'Response language (es, en)' })
  @ApiResponse({ status: 200, description: 'Team journey data' })
  @ApiResponse({ status: 502, description: 'World Cup API unavailable' })
  async getTeamJourney(@Query('lang') lang?: string): Promise<JourneyScreenModel> {
    // Delega TODO el trabajo al Service
    return this.journeyService.getJourney(lang);
  }
}

🔍 ¿Qué hace cada decorador?

Decorador ¿Qué hace? Visible en
@ApiTags('world-cup') Agrupa este endpoint en la sección "world-cup" Swagger
@Controller('world-cup') Define la ruta base del controlador Rutas HTTP
@Get('team-journey') Responde a peticiones GET en /team-journey Rutas HTTP
@ApiOperation({...}) Descripción del endpoint en Swagger Swagger
@ApiQuery({...}) Define el parámetro opcional ?lang= Swagger
@ApiResponse({...}) Documenta las posibles respuestas Swagger
@Query('lang') Extrae el parámetro ?lang= de la URL Rutas HTTP

📖 Glosario de Términos

Término Significado
API Interfaz de Programación de Aplicaciones. Un conjunto de reglas para que dos programas se comuniquen.
Backend La parte del servidor. Procesa datos, aplica lógica de negocio.
BFF Backend For Frontend. Un backend intermediario entre el frontend y otras APIs.
Controller Recibe peticiones HTTP y las delega al Service.
Decorador Anotación que agrega funcionalidad a una clase/método. Empieza con @.
Endpoint Una URL específica que acepta peticiones. Ej: GET /team-journey.
Enum Conjunto de valores constantes con nombre. Ej: colores, códigos de error.
Frontend La parte visual. Lo que el usuario ve en el navegador.
HTTP Protocolo de comunicación. GET (leer), POST (crear), PUT (actualizar), DELETE (borrar).
Interface Define la forma de un objeto. Como un molde o contrato.
Módulo Agrupa controllers y services relacionados. Como un piso del edificio.
NestJS Framework para construir backends en TypeScript.
Service Contiene la lógica de negocio. El "chef" que cocina los datos.
Swagger Herramienta de documentación interactiva para APIs.
TypeScript JavaScript con tipos. Agrega seguridad al código.
.env Archivo de variables de entorno. Configuración que cambia por entorno.
